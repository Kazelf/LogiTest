# Phase 1: Create SQL Schema Migration

## Overview

Create the initial PostgreSQL schema file for LogiTest AI's MVP data model. The migration should be plain SQL so it can run directly through `psql` without ORM setup.

## Requirements

- Functional: create `sessions`, `logs`, `personas`, `journeys`, `test_cases`, and `test_runs`.
- Functional: use UUID primary keys generated by `pgcrypto`.
- Functional: use PostgreSQL enum types for test case and test run lifecycle states.
- Functional: use `JSONB` for flexible payload, response, step, assertion, and metadata fields.
- Functional: create indexes needed by session reconstruction, ingestion lookup, and test run reporting.
- Non-functional: keep SQL readable and stable; avoid generated ORM noise.

## Architecture

The schema models the pipeline as a progression from raw imported events to generated and executed tests:

```text
sessions 1--n logs
personas 1--n journeys
journeys 1--n test_cases
personas 1--n test_cases
test_cases 1--n test_runs
```

`logs` should preserve the original record in `raw_log` while storing normalized fields for common queries. `journeys` and `test_cases` store step arrays as `JSONB` because their exact shape may evolve during MVP development.

## Implementation Steps

1. Create `logitest-ai/database/migrations/`.
2. Add `001_init_logitest_schema.sql`.
3. Enable `pgcrypto` with `CREATE EXTENSION IF NOT EXISTS pgcrypto;`.
4. Define enum types guarded by PostgreSQL-safe conditional blocks:
   - `test_case_type`: `api`, `e2e`
   - `test_case_status`: `draft`, `generated`, `approved`, `deprecated`
   - `test_run_status`: `queued`, `running`, `passed`, `failed`, `error`, `cancelled`
5. Create tables in dependency order:
   - `sessions`
   - `logs`
   - `personas`
   - `journeys`
   - `test_cases`
   - `test_runs`
6. Add indexes:
   - `idx_logs_session_occurred_at`
   - `idx_logs_trace_id`
   - `idx_logs_endpoint`
   - `idx_logs_occurred_at`
   - `idx_logs_raw_log_gin`
   - `idx_sessions_external_session_id`
   - `idx_journeys_persona_id`
   - `idx_test_cases_journey_id`
   - `idx_test_cases_status`
   - `idx_test_runs_test_case_id`
   - `idx_test_runs_status_created_at`
7. Add `database/README.md` with commands to apply and inspect the schema.

## Risk Assessment

Risk: PostgreSQL enum creation is not naturally `IF NOT EXISTS` on all versions.

Mitigation: use `DO $$ BEGIN CREATE TYPE ... EXCEPTION WHEN duplicate_object THEN NULL; END $$;`.

Risk: Fully idempotent table evolution can get complicated.

Mitigation: this is an init schema for a fresh MVP DB. Use `CREATE TABLE IF NOT EXISTS` for repeat application, and defer schema alteration workflows to a future Alembic task.
